2-网络通信

  • 本质是主机之间交换数据

socket

  • Socket 是操作系统提供的一种抽象机制，是网络通信的端点（Endpoint）
  • 它为应用程序提供了使用网络协议（主要是 TCP/IP 协议栈）进行数据发送和接收的标准编程接口（API）
  • Socket 是应用程序与网络协议栈（内核）之间的桥梁
    • 在 TCP/IP 网络中，Socket 地址由两部分组成：
      • IP 地址： 标识网络上的主机
      • 端口号（Port Number）： 标识主机上的特定进程或服务
  • 格式： (IP地址, 端口号)

以TCP为例的socket通信

服务器端流程
  1. 创建 Socket (socket()):
    • 服务器程序首先调用 socket() 函数创建一个新的 Socket。指定地址族（如 AF_INET 表示 IPv4，AF_INET6 表示 IPv6）和协议类型（如 SOCK_STREAM 表示 TCP）。
    • 结果： 获得一个 Socket 文件描述符/句柄。
  2. 绑定地址 (bind()):
    • 调用 bind() 函数将上一步创建的 Socket 绑定到一个本地 IP 地址和一个特定的端口号
    • 为什么需要绑定？ 客户端需要知道服务器的确切地址（IP+Port）才能连接。服务器通常绑定到 INADDR_ANY (所有本地IP) 和一个众所周知的端口（如 80）。
  3. 监听连接 (listen()):
    • 调用 listen() 函数，使 Socket 进入被动监听（Passive） 状态。该函数还会设置一个等待连接队列的最大长度（Backlog），用于存放等待处理的客户端连接请求。
    • 此时： Socket 准备好接受来自客户端的连接请求。
  4. 接受连接 (accept()):
    • 调用 accept() 函数。这是一个阻塞操作（默认情况下），它会等待直到有客户端发起连接请求进入等待队列。
    • 当有新的连接请求到达时，accept() 会返回一个新的 Socket 文件描述符/句柄。这个新 Socket 专门用于与这个特定的客户端进行通信。
    • 关键点： 服务器最初创建的监听 Socket (listen_sock) 只负责接受连接请求。实际的数据传输是通过 accept() 返回的新 Socket (client_sock) 进行的。服务器可以同时处理多个客户端连接，每个客户端对应一个独立的 client_sock。
  5. 数据交换 (send()/recv() or read()/write()):
    • 使用 accept() 返回的新 Socket (client_sock)，服务器可以调用 recv()（或 read()）从客户端接收数据。
    • 使用 send()（或 write()）通过 client_sock 向客户端发送数据。
    • 这个收发过程通常会持续进行，直到通信结束。
  6. 关闭连接 (close()):
    • 当通信完成，服务器调用 close() 关闭与这个客户端通信的 Socket (client_sock)。
    • 服务器程序通常会回到 accept() 步骤继续等待下一个客户端连接。
    • 最终，当服务器不再需要监听时，它会关闭最初的监听 Socket (listen_sock)。
客户端流程
  1. 创建 Socket (socket()):
    • 客户端程序调用 socket() 创建一个新的 Socket（同样指定地址族和协议类型，必须与服务器一致）。
  2. 连接服务器 (connect()):
    • 调用 connect() 函数，传入服务器端的 Socket 地址（IP 地址 + 端口号）。
    • 该函数会主动向服务器发起 TCP 连接请求（触发 TCP 三次握手过程）。
    • 结果： 如果连接成功建立，客户端的 Socket 就与服务器端的监听 Socket 建立了一个逻辑连接通道（实际上服务器端 accept() 会返回一个新 Socket 来对应这个连接）。
  3. 数据交换 (send()/recv()):
    • 使用在 connect() 中使用的同一个 Socket，客户端调用 send() 向服务器发送数据。
    • 调用 recv() 从服务器接收数据。
  4. 关闭连接 (close()):
    • 通信结束后，客户端调用 close() 关闭 Socket，终止连接
三次握手

当客户端调用 connect() 时，操作系统内核会代表应用程序发起 TCP 连接建立的“三次握手”：

  1. SYN： 客户端发送一个 SYN (Synchronize) 包给服务器，包含客户端的初始序列号。

  2. SYN-ACK： 服务器收到 SYN 包后，如果同意连接，会发送一个 SYN-ACK (Synchronize-Acknowledge) 包作为响应，包含服务器的初始序列号和对客户端 SYN 的确认。

  3. ACK： 客户端收到 SYN-ACK 包后，发送一个 ACK (Acknowledge) 包给服务器，确认服务器的 SYN。

  4. 连接建立： 服务器收到 ACK 包后，TCP 连接正式建立。此时服务器内核会将这个已建立的连接放入等待 accept() 处理的队列。当服务器调用 accept() 时，就从队列中取出这个连接，并创建新的 Socket 与之关联。

网络协议栈

  • 是一组分层组织的网络通信协议的集合

  • 核心思想： 分层（Layering） 和 封装（Encapsulation）。

    • 分层： 将复杂的网络通信过程分解成一系列相对独立、功能明确的层次。每一层都建立在下一层提供的服务之上，并为上一层提供服务。这种设计使得各层可以独立发展、修改和实现，只要层与层之间的接口保持不变。

    • 封装： 当应用程序的数据需要通过网络发送时，数据会从协议栈的顶层（应用层）开始，逐层向下传递。每一层都会在接收到的数据前（有时也在后）添加自己特定的控制信息（头部 Header，有时还有尾部 Trailer）。这个过程就像给数据包裹上了一层又一层的信封。在接收端，数据会从协议栈的底层（物理层）开始，逐层向上传递，每一层都会读取并剥离自己对应的头部（和尾部）信息，最终将原始数据交付给目标应用程序。这个过程称为解封装（Decapsulation）。

  • 目标： 实现异构系统（不同硬件、不同操作系统、不同应用程序）之间的可靠、高效、标准化的通信。

  • 协议栈最常用的两个模型是OSI 七层模型（理论参考）和TCP/IP 四层模型（实际标准）。

OSI 七层模型 (Open Systems Interconnection)

  • 理论参考模型： 由 ISO 制定，定义了网络通信的完美蓝图，清晰地划分了各层功能。虽然实际实现（尤其是 TCP/IP）不完全严格遵循它，但它仍是理解和描述网络协议栈的通用语言。

  • 各层功能： (从底层到高层)

    1. 物理层 (Physical Layer)：

      • 功能：在物理媒介（如网线、光纤、无线电波）上传输原始的比特流 (0 和 1)。
      • 设备/协议：中继器 (Repeater)、集线器 (Hub)、网卡 (NIC) 的物理部分；物理介质本身。

    2. 数据链路层 (Data Link Layer)：

      • 功能：在直接相连的两个节点（通常是同一个局域网 LAN 内）之间提供可靠的数据帧传输。负责物理寻址（MAC 地址）、差错检测（如 CRC）、流量控制（防止接收方被淹没）、介质访问控制（MAC，如 CSMA/CD 用于以太网，避免冲突）。
      • 关注：将比特流组织成帧（Frame），帧中包含源和目的 MAC 地址。
      • 设备/协议：网桥 (Bridge)、交换机 (Switch) [工作在二层]；以太网 (Ethernet - IEEE 802.3)、Wi-Fi (IEEE 802.11)、PPP、HDLC、MAC 地址。

    3. 网络层 (Network Layer)：

      • 功能：在不同网络的主机之间提供逻辑寻址（IP 地址） 和路径选择（路由）。负责将数据包（Packet） 从源主机跨网络传送到目的主机。
      • 设备/协议：路由器 (Router)；IP (IPv4, IPv6)、ICMP (用于错误报告和诊断，如 ping)、IGMP、路由协议（OSPF, BGP, RIP）。

    4. 传输层 (Transport Layer)：

      • 功能：在端到端（源应用程序进程到目的应用程序进程）提供可靠或尽力而为的数据传输服务。负责进程间通信（通过端口号 Port Number 区分同一主机上的不同应用程序）、连接管理（建立、维护、终止连接）、流量控制、差错控制（重传丢失或损坏的数据）。
      • 关注：分段（Segmentation）（将大数据块分成适合网络层传输的小段）和重组（Reassembly）。
      • 协议：TCP (Transmission Control Protocol - 可靠的、面向连接的)、UDP (User Datagram Protocol - 不可靠的、无连接的)。

    5. 会话层 (Session Layer)：

      • 功能：建立、管理和终止应用程序之间的会话（Session）。会话定义了数据交换的开始、结束和同步点。负责对话控制（全双工/半双工）、同步（在数据流中插入检查点以便出错后从检查点恢复）。
      • 注意：在实际的 TCP/IP 协议栈中，这一层的功能通常由传输层（TCP）或应用层协议实现。
      • 协议/概念：NetBIOS、RPC (Remote Procedure Call) 的部分功能、会话令牌。

    6. 表示层 (Presentation Layer)：

      • 功能：处理两个通信系统之间交换信息的语法和语义。负责数据转换（如字符编码转换 ASCII/Unicode/EBCDIC）、数据加密/解密、数据压缩/解压缩。确保应用层发出的数据能被另一台主机的应用层理解。
      • 注意：在 TCP/IP 中，这些功能通常由应用层自身处理（如 HTTPS 处理加密，应用处理 JSON/XML 格式）。
      • 概念：数据格式转换、加密 (SSL/TLS 的一部分功能可看作在此层)、压缩 (gzip)。

    7. 应用层 (Application Layer)：

      • 功能：为应用程序提供访问网络服务的接口。包含用户实际使用的各种网络应用协议。
      • 关注：应用程序特定的协议和用户接口。
      • 协议：HTTP (Web)、HTTPS (Secure Web)、FTP (文件传输)、SMTP (邮件发送)、POP3/IMAP (邮件接收)、DNS (域名解析)、SSH (安全远程登录)、Telnet (远程登录)、DHCP (动态 IP 分配)、SNMP (网络管理) 等等。
TCP/IP 四层模型 (实际标准)
  • 实践模型： 源于 ARPANET 和互联网的发展，是互联网实际运行的协议栈标准。它更简洁，将 OSI 模型的某些层进行了合并。
  • 各层功能： (从底层到高层)
    1. 网络接口层 (Network Interface Layer / Link Layer)：
      • 对应 OSI 的：物理层 + 数据链路层。
      • 功能：处理与物理网络媒介的连接、物理寻址（MAC 地址）、在本地网络链路上传输数据帧。
      • 协议/标准：以太网、Wi-Fi (802.11)、PPP、ARP (地址解析协议，将 IP 地址解析为 MAC 地址)。
    2. 网际层 (Internet Layer)：
      • 对应 OSI 的：网络层。
      • 功能：核心是 **IP (Internet Protocol)**。负责主机到主机的通信、逻辑寻址（IP 地址）、路由选择（将数据包从源主机路由到目的主机，可能跨越多个网络）。
      • 协议：IP (IPv4, IPv6)、ICMP、IGMP、路由协议（如 OSPF, BGP - 虽然实现上更复杂）。
    3. 传输层 (Transport Layer)：
      • 对应 OSI 的：传输层。
      • 功能：与 OSI 传输层功能完全相同。提供端到端的进程间通信，通过端口号区分应用，提供可靠传输 (TCP) 或尽力而为传输 (UDP)。
      • 协议：TCP、UDP。
    4. 应用层 (Application Layer)：
      • 对应 OSI 的：会话层 + 表示层 + 应用层。
      • 功能：包含了 OSI 中会话管理、数据表示转换和具体应用协议的所有功能。直接为应用程序进程提供服务。
      • 协议：所有基于 TCP/IP 的应用协议：HTTP, HTTPS, FTP, SMTP, DNS, SSH, Telnet, DHCP, SNMP 等。
